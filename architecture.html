Project Summary

The STI Assets Investor Portal is a multi-role web application that allows clients to view their investment portfolios in real time, while an internal accounts team manages and enters investment data. The frontend (HTML, CSS, JavaScript) has been completed and includes a login page, dashboard, investments page, statements, macro indicators, accounts settings, and a get help page — all sharing a single CSS and JS file with a collapsible sidebar. This document defines the complete architecture needed to make this a fully functioning, production-ready web application.

Backend — Application Server & API Layer

TO BUILD
⚡

Application Server

Node.js + Express

A RESTful API server handles all business logic between the frontend and database. It processes requests, enforces authorization, and returns structured JSON responses.

  • Node.js runtime with Express.js framework
  • JWT middleware for verifying auth tokens on every protected route
  • Role-based middleware: client vs accounts-team vs admin
  • Input validation with Joi or Zod before any DB query
  • Structured error responses (HTTP status codes + error codes)
🔐

Authentication Service

Supabase Auth

Supabase Auth handles user registration, login, password reset, and JWT token issuance — so you don't have to build any of this from scratch.

  • Email + password auth with bcrypt hashing
  • JWT access tokens (15 min expiry) + refresh tokens (7 days)
  • Password reset via email OTP link
  • Row-Level Security (RLS) policies so clients only see their own data
  • Separate staff table for accounts team roles
Endpoint Method Who Can Call Description
/auth/login POST Public Authenticates user, returns JWT + refresh token
/auth/logout POST Authenticated Invalidates refresh token, clears session
/auth/reset-password POST Public Sends password reset link to registered email
/api/portfolio GET Client Returns client's full portfolio overview and stats
/api/investments GET Client Lists all investment records for the authenticated client
/api/investments/:id/liquidate POST Client Submits a liquidation request for a specific investment
/api/investments/:id/topup POST Client Submits a top-up / additional investment request
/api/statements/:id GET Client Generates and returns PDF statement for an investment
/api/profile GET / PATCH Client Reads or updates client profile information
/api/support/ticket POST Client Creates a new support ticket
/admin/clients GET Accounts Team Lists all registered clients
/admin/clients/:id/investment POST / PATCH Accounts Team Creates or updates investment data for a client
/admin/macro POST / PATCH Admin Updates macroeconomic indicator values
💡

Supabase shortcut: Supabase auto-generates a REST API for every table you create — so for simple CRUD operations (read portfolio, update profile) you can call Supabase's built-in API directly from the frontend using the Supabase JS client, skipping the custom Node server entirely. Only build the custom Express server for complex business logic like statement PDF generation, macro data aggregation, or payment gateway integration.

Web Server

NGINX
🖥️

Static File Server (Nginx)

Nginx serves your HTML, CSS, and JS files directly. It is extremely fast at serving static content and handles thousands of concurrent connections with minimal memory.

  • Serves all HTML pages from /var/www/stiassets/
  • Gzip compression on HTML, CSS, JS (60–80% size reduction)
  • Cache-Control headers: CSS/JS cached for 1 year, HTML no-cache
  • Redirect all HTTP → HTTPS automatically
  • TLS termination using Let's Encrypt SSL certificate
⚖️

Reverse Proxy (Nginx → API)

Nginx also acts as a reverse proxy — it receives API requests on /api/* and forwards them to your Node.js Express server running internally.

  • Routes /api/* → Node.js on port 3000
  • Routes /* → static HTML files
  • Rate limiting: max 100 requests/min per IP on /auth/* routes
  • Request timeout: 30s to prevent hung connections
  • Security headers: X-Frame-Options, X-Content-Type, HSTS

CDN — Content Delivery Network

CLOUDFLARE
🌐

Cloudflare (Free Tier Recommended)

Cloudflare sits in front of everything. Nigerian investors will load the site from Cloudflare's Lagos PoP, not your origin server — making load times 3–5× faster.

  • Global edge caching of CSS, JS, images
  • DDoS protection (included free)
  • DNS management and SSL certificate (free)
  • Nigerian PoP in Lagos — fast for local users
  • Page Rules: bypass cache for /api/*, cache everything else
🔴

Redis Cache (Server-Side)

Redis caches frequently-read data server-side to reduce database load. Portfolio data, for example, only changes when the accounts team updates it — no need to query Postgres on every page load.

  • Cache portfolio data per client (TTL: 5 minutes)
  • Cache macro indicator data (TTL: 1 hour)
  • Invalidate cache when accounts team updates a client record
  • Store rate-limit counters for API endpoints
07

Load Balancer

SCALING
⚖️

Phase 1: Nginx (Built-In)

Now

At launch, Nginx itself handles load balancing by distributing requests across multiple Node.js processes using PM2's cluster mode — no separate load balancer needed.

  • PM2 cluster mode: spawns one Node.js worker per CPU core
  • Round-robin distribution across workers
  • Zero-downtime restarts when deploying new code
  • Handles hundreds of concurrent users on a single server
🔀

Phase 2: AWS ALB or DigitalOcean LB

When Scaling

When traffic grows beyond a single server, introduce a cloud load balancer to distribute requests across multiple VMs (droplets or EC2 instances).

  • Health checks every 30s — removes unhealthy servers automatically
  • Sticky sessions for WebSocket connections if added later
  • SSL termination at the load balancer level
  • Auto-scaling rules based on CPU usage thresholds

Security Components

CRITICAL
🔑

Authentication & Sessions

Replace sessionStorage with httpOnly cookies storing a JWT. HttpOnly prevents JavaScript from reading the token — defending against XSS attacks. Set SameSite=Strict and Secure flags.

🛡️

HTTPS / TLS Everywhere

All traffic must use TLS 1.2 minimum (TLS 1.3 preferred). Enforce HSTS header so browsers always use HTTPS. Cloudflare or Let's Encrypt provides the certificate for free.

🚧

Rate Limiting

Nginx rate limits /auth/login to 10 attempts per minute per IP. After 5 failed logins, lock the account for 15 minutes. Prevents brute-force attacks on client credentials.

🔒

Row-Level Security (RLS)

Supabase RLS policies ensure every database query is automatically filtered by the authenticated user's ID. A client literally cannot query another client's rows — enforced at the DB level.

🧹

Input Sanitization

All API inputs validated with Joi/Zod before hitting the database. All database queries use parameterized statements — never string concatenation. Prevents SQL injection and XSS.

📋

Audit Logging

Every write action (investment created, profile updated, liquidation requested) is logged with timestamp, user ID, and IP address in an audit_logs table. Never delete logs.

🔐

Environment Variables

Database URLs, Supabase service keys, and JWT secrets are stored in .env files on the server — never in code, never committed to Git. Use a secrets manager in production (AWS Secrets Manager or Doppler).

🌐

CORS Policy

The API server only accepts requests from your exact frontend domain (e.g., investors.stiassets.com). All other origins are rejected. Prevents unauthorized third-party sites from calling your API.

📧

2FA (Future)

Add TOTP-based two-factor authentication (Google Authenticator) for all accounts team and admin logins. Supabase Auth natively supports TOTP — can be enabled with minimal code changes.

Key Data Flow — How It All Works Together

USER JOURNEY
1

Client Signs Up

Client fills out the signup form → frontend calls Supabase Auth → Supabase creates a user record in auth.users and sends a verification email → client verifies → a matching row is created in the clients table via a database trigger.

2

Accounts Team Enters Investment Data

Accounts team logs into the admin portal using their own credentials → they search for the client → they fill in: investment type, principal, interest rate, tenure, start date, maturity date → the record is written to the investments table with their staff ID as entered_by → the client's portfolio is now populated.

3

Client Logs In and Views Dashboard

Client submits login form → frontend calls Supabase Auth → on success, a JWT is returned and stored in an httpOnly cookie → dashboard HTML loads → JavaScript calls GET /api/portfolio with the JWT cookie → API verifies the token, queries Postgres (filtered by RLS), returns the client's portfolio data → JavaScript renders the stat cards, progress bars, and chart.

4

Client Requests Liquidation

Client clicks Liquidate on an investment → modal opens → client selects reason and confirms → frontend calls POST /api/investments/:id/liquidate → API logs the transaction request in the transactions table with status "pending" → accounts team sees the request in their admin dashboard → they process it manually and update the status to "approved" or "rejected".

5

Client Logs Out

Client clicks Sign Out → frontend calls POST /auth/logout → server invalidates the refresh token in Supabase → httpOnly cookie is cleared → window.location.replace("login.html") → browser cannot go back because the cookie is gone and the auth guard on every page will catch and redirect them.
